# BrainFlow Claude Code Instructions

## Project identity

BrainFlow is a distributed large-scale bioimage processing framework.

The project goal is to let algorithm engineers write block-level algorithms while the framework handles:

- OME-Zarr reading and writing
- Dask lazy graph construction
- chunk scheduling
- GPU / CPU worker management
- progress reporting
- execution cancellation
- WebSocket state sync
- memory and GPU cache hygiene
- frontend DAG editing and monitoring

The long-term target is PB-scale data. Assume datasets are too large to fit in memory.

Before architecture-level changes, read `ARCHITECTURE.md`.

## Golden rules

### Preserve lazy execution

Never eagerly compute large data during graph construction.

Avoid in normal nodes:

- `.compute()`
- `.persist()`
- converting whole Dask arrays to NumPy
- iterating over all pixels / voxels in Python
- collecting all chunks or instances unless explicitly documented as a global barrier

Prefer:

- `dask.array.map_blocks`
- `dask.delayed`
- block-local NumPy functions
- delayed sink tasks for output nodes

The executor has two conceptual phases:

1. GraphBuilding: build lazy graph only.
2. Submit/Running: submit sink tasks to Dask and monitor progress.

Do not move compute work into GraphBuilding.

### Protect execution isolation

All runtime state must be scoped by `execution_id`.

Progress queues, WebSocket messages, state snapshots, logs, and cancellation must not leak across executions.

Queue names should be generated by helpers in `utils.progress_helper`.

### Keep algorithm nodes block-oriented

Algorithm engineers should normally implement only a NumPy block function.

For block-wise nodes, prefer subclassing `BaseBlockMapNode` and defining:

- `INPUT_TYPES`
- `RETURN_TYPES`
- `RETURN_NAMES`
- `PROCESS_BLOCK`
- optional `infer_output_dtype`
- optional skip/failure policy

Unless `ALLOW_SHAPE_CHANGE = True`, the block function must return the same shape and ndim as the input block.

### Preserve node protocol

Every user-facing backend node must be registered with:

```python
@register_node("NodeTypeName")
class MyNode:
    ...
```

Nodes should define:

- `CATEGORY`
- `DISPLAY_NAME`
- `PROGRESS_TYPE`
- `INPUT_TYPES`
- `RETURN_TYPES`
- `RETURN_NAMES`
- `FUNCTION`

Use typed Dask ports when possible:

```python
"DASK_ARRAY[uint8]"
"DASK_ARRAY[uint16]"
"DASK_ARRAY[float32]"
"DASK_ARRAY[any]"
```

Do not weaken dtype validation unless there is a clear compatibility reason.

### Respect frontend protocol

The frontend builds a ReactFlow DAG and serializes it for the backend executor.

Important frontend/backend contracts:

- `/object_info` returns node definitions.
- `/dashboard_url` returns the Dask dashboard URL.
- `/ws/run` handles execute, stop, subscribe, ping/pong.
- `execution_finished` is the authoritative terminal event.
- `done` is legacy compatibility only.
- Progress messages should include stable fields:
  - `type`
  - `taskId`
  - `executionId`
  - `progressType`
  - `progress`
  - `message`
  - `runState`
  - `progressRole`

Do not rename protocol fields without updating frontend types and handlers.

## Backend architecture notes

Important backend areas:

- `core/registry.py`: node registration and `/object_info` payload.
- `core/type_system.py`: port type and dtype compatibility.
- `core/state_manager.py`: execution sessions, logs, snapshots, subscribers.
- `core/chunk_policy.py`: chunk recommendation and rechunk policy.
- `services/dask_service.py`: Dask LocalCluster lifecycle and worker configuration.
- `services/executor.py`: DAG validation, lazy graph building, sink submission, progress monitoring, cleanup.
- `nodes/base.py` or `block_map.py`: block map node abstraction.
- `nodes/ome_zarr_flow.py`: OME-Zarr reader/writer.
- `nodes/cellpose_node.py`: Cellpose segmentation and model cache.
- `nodes/instance_nodes.py`: per-chunk instance extraction.
- `nodes/post_processing_nodes.py`: distributed statistics and global merge barriers.
- `utils/progress_helper.py`: Dask queue progress helpers.
- `utils/memory_monitor.py`: process, worker, and GPU memory snapshots.

## Frontend architecture notes

Important frontend areas:

- `App.tsx`: page shell.
- `FlowContext.tsx`: global editor/execution state composition.
- `FlowEditor.tsx`: ReactFlow canvas.
- `DynamicNode.tsx`: dynamic node rendering based on backend node definitions.
- `Sidebar.tsx`: node library.
- `Header.tsx`: run/stop/save/load/dashboard controls.
- `BottomPanel.tsx`: logs, status, progress, chunk counters.
- `useFlowEngine.ts`: backend protocol, WebSocket, graph serialization, progress handling.
- `useFlowOperations.ts`: node/edge operations.
- `useWorkflows.ts`: workflow persistence.
- `portTypes.ts`: frontend port compatibility display.

When changing backend message schema, update frontend types and `useFlowEngine.ts`.

## Dask, chunks, and PB-scale safety

Always reason about:

- chunk shape
- chunk count
- task graph size
- block memory footprint
- worker memory limit
- spill directory
- unmanaged memory growth
- GPU VRAM pressure
- repeated upstream computation
- final global barriers

For any operation that gathers global data, document the scaling limit and expected memory behavior.

Global barriers are allowed only when needed and clearly bounded, such as DaskStats merge-map construction.

## GPU and Cellpose rules

Do not reload Cellpose models for every chunk.

Respect worker-level model cache and reference counting.

Avoid high-frequency `torch.cuda.empty_cache()` or `torch.cuda.synchronize()` inside every chunk unless the user explicitly asks for a memory-debug mode.

Cluster stop or execution cleanup should clear worker-side caches where appropriate.

## OME-Zarr rules

Readers should:

- validate paths
- preserve metadata where possible
- create Dask arrays with intended chunks directly
- avoid unnecessary rechunking

Writers should:

- build delayed write plans
- avoid compute during graph construction
- write metadata only after all block writes finish
- handle boundary chunks correctly
- use actual chunk metadata for region slices, not only `arr.chunksize`

## Progress rules

Progress types:

- `CHUNK_COUNT`: chunk-level progress via queue.
- `STATE_ONLY`: status text only.
- `STAGE_BASED`: phase/stage progress via stage queue.

Progress reporting must be execution-scoped.

Use `report_progress` and `report_stage_progress`; do not create ad-hoc queue names.

## Error handling

Prefer explicit actionable errors.

Good errors include:

- node id
- node type
- input name if relevant
- expected type/shape/dtype
- actual type/shape/dtype
- suggested fix

Do not silently swallow exceptions unless a node explicitly has a failure policy like `zeros_like`.

## Testing and validation

When changing backend logic, add or suggest minimal tests.

Useful test targets:

- DAG cycle detection
- graph type validation
- dynamic return type resolution
- block node dtype and shape validation
- chunk helper behavior
- OME-Zarr writer edge chunks
- progress queue naming with execution_id
- cancellation state transitions
- state snapshot recovery
- frontend graph serialization
- frontend progress message handling

Use synthetic small Dask arrays and temporary Zarr stores. Do not require real large datasets in tests.

## How to work

For non-trivial tasks:

1. Inspect relevant files first.
2. State a short plan.
3. Identify risks before editing.
4. Make the smallest safe change.
5. Preserve existing frontend/backend protocol unless asked to change it.
6. Run or propose focused validation.
7. Summarize changed files, behavior, tests, and remaining risks.

Do not perform broad rewrites without explicit approval.

## Security and data safety

Do not read or print secrets.

Avoid reading large datasets, model checkpoints, generated `.zarr` stores, or private `.env` files unless the user explicitly asks.

Never hard-code API keys, file-system credentials, or machine-specific absolute paths.

## Done definition

A change is complete only when:

- large-data laziness is preserved
- execution_id isolation is preserved
- frontend/backend protocol remains consistent
- errors are actionable
- relevant validation is provided
- performance and memory risks are noted
